# Sparkamp — macOS Port Requirements

> **Branch:** `OSX-port`  
> **Audience:** Claude (primary implementer), human reviewers  
> **Status:** Planning — no code written yet

---

## 1. Philosophy & Goals

The macOS port must feel like a first-class native application, not a Linux app bolted onto macOS. The GTK4 frontend is not available on macOS; it is replaced with a SwiftUI frontend. The Rust core — the audio engine, playlist model, config persistence, media library, ID3 editor, and shuffle/repeat logic — stays Rust and is shared byte-for-byte between the Linux and macOS builds. No features are dropped.

The TUI (`sparkamp` without `--ui`) is already cross-platform and will work on macOS without any changes. The macOS GUI port is separate from and does not disturb the TUI.

Key constraints:

- **No new audio backend.** GStreamer stays (available via Homebrew, bundleable).
- **No forked core.** Platform-specific code in `src/` is minimised — ideally zero for the audio/model/config layer.
- **Iterative milestones.** Each milestone is self-contained: implement → test → merge to `main` → start next milestone. No milestone depends on later ones being finished.
- **macOS minimum target:** macOS 14 (Sonoma).
- MacOS port must compile to a standalone .app file wit any dependencies either included or triggered to be installed with the app.

## 2. Architecture

### 2.1 Current structure

```
src/               ← Core Rust (engine, model, config, controller,
                     media_library, shuffle, id3_editor, dedupe,
                     granite, duration_cache, duration_probe, skin)
frontends/
  gtk/             ← GTK4 GUI (Linux only)
  tui/             ← Terminal UI (cross-platform, no changes needed)
```

### 2.2 Target structure after port

```
src/               ← Core Rust (unchanged for existing features)
  ffi.rs           ← NEW: extern "C" functions that wrap the public
                     Rust API for consumption by Swift
frontends/
  gtk/             ← GTK4 GUI (Linux, no changes from port work)
  tui/             ← Terminal UI (no changes)
  macos/           ← NEW: macOS-specific Rust bridge crate
    Cargo.toml     ← crate-type = ["staticlib"], depends on core
    build.rs       ← calls cbindgen to regenerate include/sparkamp.h
    src/lib.rs     ← re-exports src/ffi.rs symbols
  SparkampMac/     ← NEW: Swift/SwiftUI application (Xcode project)
    SparkampMac.xcodeproj
    SparkampCore/  ← bridging header + Swift wrapper around C API
    Sources/       ← SwiftUI views and app delegates
include/
  sparkamp.h       ← AUTO-GENERATED by cbindgen from src/ffi.rs
```

### 2.3 Build process on macOS

```bash
# 1. Install dependencies
brew install gstreamer gst-plugins-base gst-plugins-good \
             gst-plugins-bad gst-plugins-ugly

# 2. Build the Rust static library
cargo build --release -p sparkamp-macos

# 3. Open and build the Swift app (Xcode)
open frontends/SparkampMac/SparkampMac.xcodeproj
# or: xcodebuild -project frontends/SparkampMac/...
```

The Swift app links `libsparkamp_macos.a` (produced by step 2) and imports `include/sparkamp.h` via a bridging header. GStreamer is linked via pkg-config inside the build script.

### 2.4 Platform-conditional compilation in `src/`

The core already compiles on macOS; the only change needed in `src/main.rs` is:

```rust
#[cfg(target_os = "linux")]
fn main_gtk(args: ...) { ... }   // existing GTK entry point

#[cfg(target_os = "macos")]
fn main_gtk(_: ...) {
    eprintln!("Use the Sparkamp.app bundle for the GUI on macOS.");
    std::process::exit(1);
}
```

Nothing else in `src/` needs `#[cfg(target_os)]` — the audio engine, model, config, and all business logic are already fully cross-platform.

### 2.5 FFI design principles

- `src/ffi.rs` owns one opaque `SparkampContext` heap object that Swift holds as `*mut c_void`.
- All strings cross the boundary as null-terminated UTF-8 (`*const c_char` / `*mut c_char`).
- All returned strings that Swift must free are freed via `sparkamp_free_string()`.
- All callback registrations use `unsafe extern "C" fn` pointers; Swift provides `@convention(c)` closures.
- State changes that the UI needs to know about (EOS, error, position update, spectrum data) are reported via registered callback, not polling — Swift sets up a dispatch queue and the callback posts to the main queue.

## 3. Core Refactoring (pre-milestone work)

These changes are a prerequisite for all milestones. They are done once on the `OSX-port` branch before Milestone 1 begins, and merged as the first commit.

### 3.1 New crate: `frontends/macos` (Rust bridge)

`Cargo.toml`:
```toml
[package]
name = "sparkamp-macos"
edition = "2024"

[lib]
name = "sparkamp_macos"
crate-type = ["staticlib"]

[dependencies]
sparkamp = { path = "../.." }     # the main crate's pub types

[build-dependencies]
cbindgen = "0.26"
```

`build.rs`:
```rust
fn main() {
    let crate_dir = std::env::var("CARGO_MANIFEST_DIR").unwrap();
    cbindgen::Builder::new()
        .with_crate(&crate_dir)
        .with_language(cbindgen::Language::C)
        .generate().unwrap()
        .write_to_file("../../include/sparkamp.h");
}
```

### 3.2 `src/ffi.rs`

Added incrementally with each milestone; starts empty. Each function is documented with its expected call thread and ownership rules. The file is compiled only when building `sparkamp-macos` (use `#[cfg(feature = "ffi")]` or just leave it as dead code guarded behind the macos crate re-exporting it).

### 3.3 `Cargo.toml` (workspace root)

Add the macos bridge crate to the workspace:
```toml
[workspace]
members = [
    ".",
    "frontends/macos",    # macOS bridge
]
```

Add `cbindgen` to dev-dependencies at workspace level.

### 3.4 Xcode project skeleton

Create `frontends/SparkampMac/SparkampMac.xcodeproj` with:
- App target: `SparkampMac`
- Deployment target: macOS 14.0
- Swift language version: 5.9+
- Bridging header: `SparkampCore/sparkamp_bridge.h` (includes `sparkamp.h`)
- Build phase: runs `cargo build --release -p sparkamp-macos` before compilation
- Linker flags: add the `.a` path, link GStreamer frameworks via pkg-config output

#### 1.3 Keyboard shortcuts (macOS equivalents)

The GTK frontend uses bare keys (`z`, `x`, `c`, etc.) via a key capture controller. On macOS, bare keys conflict with text inputs; use the same keys but only when the playlist or main window has focus and no text field is editing.

| Key | Action | macOS notes |
|-----|--------|-------------|
| `z` | Previous | Same |
| `x` | Play | Same |
| `c` | Pause/Resume | Same |
| `v` | Stop | Same |
| `b` | Next | Same |
| `r` | Cycle repeat | Same |
| `s` | Toggle shuffle | Same |
| `←` / `→` | Seek ±5 s | Same |
| `-` / `=` | Volume −5% / +5% | Same as GTK (not `[`/`]` which are TUI only) |
| `p` | Toggle playlist visibility | Same |
| `⌘O` | Add file(s) | macOS standard (replaces bare `n`) |
| `Del` / `⌫` | Remove selected playlist track | Both keys |

#### 1.4 Persist state between sessions

On shutdown:
- Call `sparkamp_save_config()` → writes to `~/Library/Application Support/sparkamp/config.toml` (handled automatically by the `dirs` crate in Rust)
- Save playlist to `~/Library/Application Support/sparkamp/last_playlist.toml`
- Save window position/size via `@AppStorage` or `UserDefaults`

On startup:
- Call `sparkamp_load_config()`
- Load last playlist (Rust side already handles this in `Playlist::load_last()`)

#### 1.5 Drag-and-drop

Support dropping audio files (mp3, flac, ogg, opus, wav, aac, m4a) and folders onto the main window or the playlist. Folders are added recursively (same logic as `is_audio_file()` in `src/model.rs`).

#### 1.6 Duration probing

The Rust background probing system (`src/duration_probe.rs`) works on macOS via `rayon`. Expose:
```c

// fires position_callback with updated duration when done
```

Linux vs macOS Skin System: Delta Analysis & Plan
Current Delta
1. Color variable mismatches between the two style_dark.css and macOS dark defaults

The portable :root { --sparkamp-* } block exists in style_dark.css, but several values differ from what macOS uses in its built-in dark skin:

Variable    macOS dark default    Linux style_dark.css
--sparkamp-playlist-current-text    #00ccff (cyan)    #00ff88 (green)
--sparkamp-accent    #00aaff    not defined; falls back to GTK @accent_bg_color
2. Linux uses GTK dynamic variables; macOS uses hardcoded hex

style_dark.css references @accent_bg_color, @accent_fg_color, and @headerbar_bg_color — GTK CSS-only tokens that don't exist in macOS's CSS parser. The macOS CSSParser ignores unknown @ tokens but the intent — "use the system accent color" — is completely lost on macOS. Since macOS uses hardcoded hex values, the equivalent portable fix is to pick a single accent hex value for the skin file and use that everywhere.

3. GTK-specific widget rules below the separator comment are macOS-invisible

The skin file has a separator comment /* === GTK-specific === */ (or similar) after the :root block. Everything below it is ignored by macOS. Currently this section contains critical layout rules for:

entry (text input fields)
columnview, listview (the ML table and playlist list)
notebook (tab widget for ML Files/Playlists)
popover modelbutton (context menu items)
button.text-button, button.suggested-action
scale.eq-scale (equalizer sliders)
.np-frame (the now-playing area border)
.unscanned-row (ML rows where metadata hasn't loaded yet — black background on Linux)
macOS renders these through SwiftUI native components and reads only the :root CSS variables. So as long as variables are defined correctly, all the widget styling flows through automatically on macOS. The GTK-specific section is fine to keep as-is.

4. macOS-only variables are undefined in the Linux skin

macOS reads several variables that style_dark.css doesn't define:

--sparkamp-logo-text / --sparkamp-logo-subtext — used for the SPARKAMP/version text overlay on the player logo. Linux uses a raster PNG and ignores these.
--sparkamp-lcd-background — used for the time display LCD area
--sparkamp-mode-btn-text — toolbar icon color
The GTK frontend ignores undefined custom properties, so this is silent. But a truly shared skin file needs these defined so macOS renders them correctly.

5. style.css (light skin) has no :root block at all

style.css is the legacy light-mode file with hardcoded color values throughout. It has no portable :root block, so macOS can't use it as a skin at all. It would need to be converted to the same dual-format structure as style_dark.css.

6. .unscanned-row vs Swift scanned bool

Linux colors unscanned ML rows using a .unscanned-row CSS class (black background). macOS uses the scanned Bool on MLTrack to dim the text to playlistDurationText color instead. The approaches are equivalent in intent but different in mechanism — no shared CSS variable needed here, but the behavior should be documented.

Plan (no code changes)
Step 1 — Unify the accent color

Replace all @accent_bg_color / @accent_fg_color references in style_dark.css with a concrete --sparkamp-accent hex value (e.g. #0099ff). Add a --sparkamp-accent-fg: #ffffff companion. In the GTK-specific section, reference var(--sparkamp-accent) instead of the GTK token. This makes the skin file self-contained for both frontends.

Step 2 — Fix --sparkamp-playlist-current-text

Change the Linux dark skin value from #00ff88 to #00ccff to match macOS. (Or decide macOS should use #00ff88 — pick one canonical value and apply it to both.) One shared source of truth.

Step 3 — Add the macOS-only variables to the :root block

Add --sparkamp-logo-text, --sparkamp-logo-subtext, --sparkamp-lcd-background, --sparkamp-mode-btn-text to the :root block in style_dark.css (and eventually style.css). GTK ignores extras silently; macOS needs them. Dark skin values to add:

--sparkamp-logo-text:     #e0e0e0;
--sparkamp-logo-subtext:  #888888;
--sparkamp-lcd-background: #0a0a0a;
--sparkamp-mode-btn-text: #aaaaaa;
Step 4 — Convert style.css to dual-format

Audit all hardcoded hex values in style.css, extract them into a :root { --sparkamp-* } block mirroring the dark skin's variable names but with light-mode values, then replace inline colors with var(--sparkamp-*) references. Add the same macOS-only variables with light values.

Step 5 — Document the separator convention in the skin format spec

Add a comment block at the top of both CSS files (or a SKINS.md) explaining:

Everything inside :root { } is portable — read by both macOS CSSParser and GTK CSS engine
GTK-specific rules (widget selectors) go after /* === GTK-specific === */ and are silently ignored by macOS
macOS-only variables should still be defined in :root — GTK ignores unknown custom properties
No @accent_bg_color or other GTK CSS level 4 tokens in the :root block
After steps 1–5, the same .css file can be dropped into either frontend's skin directory and render correctly on both platforms with no modifications.
